# v1.9 Roadmap — extensibility: component additions, a Tailwind-like extension surface, and template/block distribution

> Status: **in progress** — opened 2026-06-18. CEO direction (2026-06): "add components, distribute templates, extend Tailwind-like." Decisions: [ADR-0020](../adr/0020-distribution-channels.md) (distribution), [ADR-0021](../adr/0021-extension-contract.md) (extension contract).
> Predecessors: [v1.0](./v1.0.md) · … · [v1.7](./v1.7.md) · [v1.8](./v1.8.md) (all shipped).

## Goal

Grow the design system from a fixed primitive set into an **extensible** one, without discarding what ships today. Three layers, additive:

1. **Keep the npm primitives as the contract layer** — `@willink-labs/*` stay lockstep-versioned, import-and-run.
2. **A Tailwind-like extension surface** — a published `@theme` override contract, the dark contract as a named `@custom-variant`, and CSS-first `@layer components` classes (`.wl-btn`/`.wl-card`/`.wl-input`); defer the JS `@plugin`.
3. **Template / block distribution** — a self-hosted shadcn-compatible registry (`apps/registry`) for copy-to-own blocks (whose `registryDependencies` point at the npm primitives), plus GitHub template repos for whole-app starters.

Everything is additive under [ADR-0010](../adr/0010-semver-policy.md): new components are MINOR npm bumps; the registry and templates are new public-MIT surfaces, not a rewrite.

## Scope

| Area | What ships |

## Release shape

Components ship lockstep as MINOR npm bumps as they land (no single big-bang cut). The registry and template repos version independently of the npm packages. Verification per [ADR-0012](../adr/0012-release-verification-policy.md): every component PR runs the Layer-0 gate + the component checklist; the registry adds a build-time `'use client'` guard analogous to `check-dist-use-client.mjs`.

This is an open, multi-cycle effort — the backlog is tracked as `loop:approved` issues and consumed by the supervised/automated dev loop.
