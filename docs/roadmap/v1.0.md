# v1.0.0 Release Roadmap

> Status: **shipped** — v1.0.0 released 2026-05-17 ([PR #22](https://github.com/willink-oss/willink-design-system/pull/22), tags `v1.0.0` / `flutter-v1.0.0`). Updated: 2026-06-11.
> Tracking issue: [#16](https://github.com/willink-oss/willink-design-system/issues/16). Phase issues: [#10](https://github.com/willink-oss/willink-design-system/issues/10) – [#15](https://github.com/willink-oss/willink-design-system/issues/15).

## Outcome (recorded after release)

The plan below is kept as written for the historical record. The release deviated from it in four ways:

1. **Scope cut — Storybook (Phase 9.3).** Not shipped. [#12](https://github.com/willink-oss/willink-design-system/issues/12) stays open and carries over to [v1.1](./v1.1.md).
2. **Scope cut — Flutter Phase-7 expansion (Phase 9.4).** None of the four components shipped; `willink_theme` froze at the existing five components and bumped to 1.0.0 as an API-freeze marker only. [#13](https://github.com/willink-oss/willink-design-system/issues/13) stays open and carries over to [v1.1](./v1.1.md).
3. **Scope addition — `@willink-labs/css-tokens`.** A fifth package (framework-agnostic static CSS variables) was added in 0.13.0 ([ADR-0009](../adr/0009-css-tokens-package.md)) and froze at 1.0.0 with the rest. The lockstep set is now five npm-side packages, not three.
4. **Version ladder.** Actual path was 0.9.0 (state tokens) → 0.10.0 (gradient tokens) → 0.11.0 (OKLCH brand scale) → 0.12.0 (motion tokens, Phase 9.1) → 0.13.0 (reduced-motion + a11y matrix + css-tokens, Phase 9.2) → 1.0.0 — not the three-step ladder planned below. ADR numbering shifted accordingly: semver policy landed as ADR-0010 and Flutter versioning as ADR-0011 (0009 was taken by css-tokens).

Phases 9.1, 9.2, 9.5, and 9.6 shipped as planned. See [`docs/MIGRATION-0.8-to-1.0.md`](../MIGRATION-0.8-to-1.0.md) for the adopter-facing summary.

## Goal

The current packages — `@willink-labs/tokens`, `@willink-labs/tailwind-preset`, `@willink-labs/react` at 0.8.0, and `willink_theme` at 0.5.0 — have completed the OSS-public phase (Phase 8) and the single-brand baseline pivot (0.8.0). The next milestone, **v1.0.0**, means an API freeze plus the supporting documentation required for adopters to upgrade with confidence.

This document describes the public roadmap from 0.8.0 to 1.0.0 so that contributors and consumers can see what is in scope, what is intentionally deferred, and how the releases ladder up.

## Guiding principles

- **API freeze first, features second.** The 24 React components and 5 Flutter components already cover the surface. v1.0 stabilizes that surface; it does not ship a large new component wave.
- **Independent Flutter cadence is OK.** `willink_theme` may version independently of the three npm packages (this is already the case at 0.5.0 vs 0.8.0). The Phase 9.5 ADR formalizes this.
- **No tooling without same-PR content.** The Phase 0 cultural rule (see `README.md` and `scripts/check-guardrails.mjs`) still applies — introducing Storybook must come with stories in the same PR.

## In scope for v1.0

| Area | What ships |
|---|---|
| API freeze | Audit every export in `packages/react/src/index.ts`; lock prop names, types, and compound exports |
| Accessibility | A WCAG 2.1 AA compliance matrix per component + `prefers-reduced-motion` coverage |
| Motion | Semantic motion tokens (role-based: `motion.modal-enter`, `motion.toast`, …) wired into components |
| Documentation | Storybook 9 with stories for all 24 React components, coexisting with `apps/playground` |
| Flutter parity | Four new Flutter components: `WillinkTabBar`, `WillinkBottomSheet`, `WillinkSnackBar`, `WillinkProgressIndicator` |
| Governance | ADR for semver policy (post-1.0) and ADR for the Flutter-independent versioning convention |
| Migration | `docs/MIGRATION-0.8-to-1.0.md` for adopters |

## Out of scope for v1.0 (deferred to v1.1+)

| Item | Note |
|---|---|
| Dark mode | Light-only ships first to keep the API surface fixed. Dark mode is its own design cycle. |
| WordPress package | Mentioned in the README as a future consumer group. v1.1+ candidate. |
| `FormField` compound export | The current `Input` + `Label` + Radix combination is sufficient. A compound API deserves its own design cycle. |
| Flutter `Checkbox` / `Switch` / `RadioGroup` wrappers | Material 3 native controls already brand-respect via `ColorScheme`. |
| `Popover` / `Combobox` / `Breadcrumb` / `Stepper` (React) | Not planned for v1.0. Open a GitHub Discussion to propose. |

## Release ladder

| Release | Phases included | Character |
|---|---|---|
| **0.8.0** (shipped) | Single-brand baseline pivot — brand axis machinery removed, consumer overrides via `:root` CSS variables | — |
| **0.9.0** | Phase 9.1 (API audit) + Phase 9.2 (a11y matrix, motion tokens, reduced-motion coverage) | Documentation + low-risk additive changes. Pre-1.0 minor-breaking allowance still applies. |
| **0.10.0** | Phase 9.3 (Storybook bootstrap + stories) + Phase 9.4 PR1–2 (Flutter `WillinkTabBar` + `WillinkBottomSheet`) | Tooling-guardrail validation + Flutter expansion. |
| **1.0.0** | Phase 9.4 PR3–4 (`WillinkSnackBar`, `WillinkProgressIndicator`) + Phase 9.5 (migration / ADRs) + Phase 9.6 (release cut) | **No new React components.** Stabilization, documentation, and the version bump. |

## Phases

Each phase is tracked as its own GitHub Issue with a checklist. The tracking issue links to all six.

### Phase 9.1 — Public API audit and freeze prep (target: 0.9.0) — [#10](https://github.com/willink-oss/willink-design-system/issues/10)

Audit every exported symbol in `packages/react/src/index.ts` and lock its surface. Decide the `FormField` question (recommended: defer to v1.1). Promote motion primitives in `packages/tokens/src/primitive.json` to role-based semantic tokens in `semantic.json` and rewire the components that currently hardcode durations.

Template: `packages/react/src/components/button/button.tsx` (forwardRef + CVA + `cn` + Props type export).

### Phase 9.2 — A11y matrix and motion contract (target: 0.9.0) — [#11](https://github.com/willink-oss/willink-design-system/issues/11)

Publish a per-component WCAG 2.1 AA compliance table and a motion contract table tied to the semantic tokens introduced in 9.1. Add `motion-reduce:transition-none` to the components currently relying on Tailwind defaults (`Dialog`, `Sheet`, `Toast`, `Accordion`, `Tooltip`) to close the WCAG 2.3.3 gap. Write ADR-0008 documenting the motion token design.

### Phase 9.3 — Storybook 9 introduction (target: 0.10.0) — [#12](https://github.com/willink-oss/willink-design-system/issues/12)

Bootstrap `apps/storybook/` alongside the existing `apps/playground/`. The bootstrap PR must include stories for at least three components to satisfy the same-PR-content rule. Subsequent PRs add stories for the remaining components in batches of five. Enable `@storybook/addon-a11y`; cross-link with the Phase 9.2 matrix. Confirm Storybook 9 + Tailwind v4 + React 19 compatibility before committing to the bootstrap PR.

### Phase 9.4 — Flutter Phase-7 expansion (target: 0.10.0 → 1.0.0) — [#13](https://github.com/willink-oss/willink-design-system/issues/13)

Four PRs, one per component, each bumping `willink_theme` independently (formalized in ADR-0010 from Phase 9.5):

1. `WillinkTabBar` — parity with React `Tabs`, common mobile tab-navigation pattern.
2. `WillinkBottomSheet` — parity with React `Sheet`, wraps Material 3 `ModalBottomSheet` with theme-aware styling.
3. `WillinkSnackBar` — parity with React `Toast`, theme-aware wrapper over `ScaffoldMessenger.showSnackBar`.
4. `WillinkProgressIndicator` — parity with React `Progress`, determinate and indeterminate variants.

Template: `packages/flutter_theme/lib/src/components/willink_button.dart`. Each component requires `flutter analyze` clean, the standard 7-test pattern, and a clean `dart pub publish --dry-run`.

### Phase 9.5 — Migration guide and SemVer ADRs (target: 1.0.0 prep) — [#14](https://github.com/willink-oss/willink-design-system/issues/14)

- `docs/adr/0009-semver-policy.md` — define what constitutes MAJOR / MINOR / PATCH after 1.0.0.
- `docs/adr/0010-flutter-versioning.md` — formalize the convention that `willink_theme` versions independently of the three npm packages.
- `docs/MIGRATION-0.8-to-1.0.md` — adopter guide, cross-checked against `git diff v0.8.0 HEAD -- packages/react/src/index.ts`.
- Update `CONTRIBUTING.md` and the `README.md` roadmap section.

### Phase 9.6 — v1.0.0 release cut — [#15](https://github.com/willink-oss/willink-design-system/issues/15)

- Bump the three npm packages to 1.0.0 in a single release PR with CHANGELOG entries linking to the migration guide. `willink_theme` bumps to 1.0.0 in the same PR (the version coincidence is a marker, not a coupling commitment going forward).
- Tag `v1.0.0` (npm packages) and `flutter-v1.0.0` (pub.dev) so each workflow publishes from its own source of truth.
- Re-enable `--provenance` in `.github/workflows/publish.yml` now that the repository is public (resolves the note in ADR-0007).
- Draft GitHub Release notes; open a GitHub Discussions post for the announcement.

## Parallelism

```
                  ┌─ 9.2 (a11y / motion)  ─┐
9.1 (API freeze) ─┤                        ├─→ 0.9.0
                  └─ 9.4 PR1 (TabBar)     ─┘     │
                                                 ├─ 9.3 (Storybook)        ─┐
                                                 └─ 9.4 PR2 (BottomSheet)  ─┤
                                                                            ├─→ 0.10.0
                                                  9.4 PR3 (SnackBar)       ─┤
                                                  9.4 PR4 (Progress)       ─┤
                                                  9.5 (migration / ADRs)   ─┤
                                                                            └─→ 9.6 → v1.0.0
```

- **Strictly sequential:** 9.1 → 9.2 (motion token promotion is a prerequisite); 9.5 → 9.6 (migration doc must land before the tag).
- **Parallel-safe:** 9.2 ‖ 9.4-PR1; 9.3 ‖ 9.4-PR2.

## How to contribute

- Each phase is a GitHub Issue with a checklist. Pick one up, open a PR that references the issue, and tick boxes as you go.
- Keep `pnpm guardrails` green on every PR.
- Storybook PRs must include stories in the same PR — no empty tooling commits.
- Flutter-only PRs bump `willink_theme` alone; do not touch the npm packages unless tokens or preset CSS change.
- Open a GitHub Discussion if a deferred item (dark mode, WordPress, FormField, etc.) needs to be reconsidered for v1.0 scope — proposals are welcome.
