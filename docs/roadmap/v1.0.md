# v1.0.0 Release Roadmap

> Status: planning. Updated: 2026-05-16.
> Tracking issue: [#16](https://github.com/willink-oss/willink-design-system/issues/16). Phase issues: [#10](https://github.com/willink-oss/willink-design-system/issues/10) – [#15](https://github.com/willink-oss/willink-design-system/issues/15).

## Goal

The current packages — `@willink-labs/tokens`, `@willink-labs/tailwind-preset`, `@willink-labs/react` at 0.7.1, and `willink_theme` at 0.4.0 — have completed the OSS-public phase (Phase 8). The next milestone, **v1.0.0**, means an API freeze plus the supporting documentation required for adopters to upgrade with confidence.

This document describes the public roadmap from 0.7.1 to 1.0.0 so that contributors and consumers can see what is in scope, what is intentionally deferred, and how the releases ladder up.

## Guiding principles

- **API freeze first, features second.** The 24 React components and 5 Flutter components already cover the consumer surface. v1.0 stabilizes that surface; it does not ship a large new component wave.
- **Lockstep with one documented exception.** The three npm packages continue to bump together. `willink_theme` may bump independently when the change is Flutter-only — this exception is to be formalized in an ADR before the v1.0 cut.
- **No tooling without same-PR content.** The Phase 0 cultural rule (see `README.md` and `scripts/check-guardrails.mjs`) still applies — introducing Storybook must come with stories in the same PR.

## In scope for v1.0

| Area | What ships |
|---|---|
| API freeze | Audit every export in `packages/react/src/index.ts`; lock prop names, types, and compound exports |
| Accessibility | A WCAG 2.1 AA compliance matrix per component + `prefers-reduced-motion` coverage |
| Motion | Semantic motion tokens (role-based: `motion.modal-enter`, `motion.toast`, …) wired into components |
| Documentation | Storybook 9 with stories for all 24 React components, coexisting with `apps/playground` |
| Flutter parity | Four new Flutter components: `WillinkTabBar`, `WillinkBottomSheet`, `WillinkSnackBar`, `WillinkProgressIndicator` |
| Governance | ADR for semver policy (post-1.0) and ADR for the lockstep exception |
| Migration | `docs/MIGRATION-0.7-to-1.0.md` for adopters |

## Out of scope for v1.0 (deferred to v1.1+)

| Item | Reason |
|---|---|
| Dark mode | Light-only ships first to keep the API surface fixed. Dark mode is its own design cycle. |
| WordPress package | Mentioned in the README as a future consumer group; no implementation exists yet. v1.1+ candidate. |
| `FormField` compound export | The current `Input` + `Label` + Radix combination is sufficient. A compound API deserves its own design cycle. |
| Flutter `Checkbox` / `Switch` / `RadioGroup` wrappers | Material 3 native controls already follow the brand axis via `ColorScheme`. Wrapping ROI is low. |
| `Popover` / `Combobox` / `Breadcrumb` / `Stepper` (React) | Not requested by current consumers; revisit when concrete demand surfaces. |

## Release ladder

| Release | Phases included | Character |
|---|---|---|
| **0.8.0** | API audit, a11y matrix, motion token promotion, reduced-motion coverage | Documentation + low-risk additive changes. Pre-1.0 minor-breaking allowance still applies. |
| **0.9.0** | Storybook bootstrap + first stories, Flutter `WillinkTabBar` and `WillinkBottomSheet` | Validates the tooling-guardrail handling and exercises the lockstep exception for Flutter-only releases. |
| **1.0.0** | Remaining Flutter components, migration guide, semver / lockstep ADRs, release cut | **No new React components.** Stabilization, documentation, and the version bump. |

Dependabot PRs that are open at planning time should be batched and merged before the 0.8.0 work starts, to avoid lockfile drift during the freeze.

## Phases

Each phase is tracked as its own GitHub Issue with a checklist. The tracking issue links to all six.

### Phase 9.1 — Public API audit and freeze prep (target: 0.8.0) — [#10](https://github.com/willink-oss/willink-design-system/issues/10)

Audit every exported symbol in `packages/react/src/index.ts` and lock its surface. Decide the `FormField` question (recommended: defer to v1.1). Promote motion primitives in `packages/tokens/src/primitive.json` to role-based semantic tokens in `semantic.json` and rewire the components that currently hardcode durations.

Template: `packages/react/src/components/button/button.tsx` (forwardRef + CVA + `cn` + Props type export).

### Phase 9.2 — A11y matrix and motion contract (target: 0.8.0) — [#11](https://github.com/willink-oss/willink-design-system/issues/11)

Publish a per-component WCAG 2.1 AA compliance table and a motion contract table tied to the semantic tokens introduced in 9.1. Add `motion-reduce:transition-none` to the components currently relying on Tailwind defaults (`Dialog`, `Sheet`, `Toast`, `Accordion`, `Tooltip`) to close the WCAG 2.3.3 gap. Write ADR-0008 documenting the motion token design.

### Phase 9.3 — Storybook 9 introduction (target: 0.9.0) — [#12](https://github.com/willink-oss/willink-design-system/issues/12)

Bootstrap `apps/storybook/` alongside the existing `apps/playground/`. The bootstrap PR must include stories for at least three components to satisfy the same-PR-content rule. Subsequent PRs add stories for the remaining components in batches of five. Enable `@storybook/addon-a11y`; cross-link with the Phase 9.2 matrix. Confirm Storybook 9 + Tailwind v4 + React 19 compatibility before committing to the bootstrap PR.

### Phase 9.4 — Flutter Phase-7 expansion (target: 0.9.0 → 1.0.0) — [#13](https://github.com/willink-oss/willink-design-system/issues/13)

Four PRs, one per component, each bumping `willink_theme` independently (lockstep exception, documented in ADR-0010 from Phase 9.5):

1. `WillinkTabBar` — parity with React `Tabs`, high need for clubhouse-style navigation.
2. `WillinkBottomSheet` — parity with React `Sheet`, wraps Material 3 `ModalBottomSheet` with brand-aware styling.
3. `WillinkSnackBar` — parity with React `Toast`, brand-aware wrapper over `ScaffoldMessenger.showSnackBar`.
4. `WillinkProgressIndicator` — parity with React `Progress`, determinate and indeterminate variants.

Template: `packages/flutter_theme/lib/src/components/willink_button.dart`. Each component requires `flutter analyze` clean, the standard 7-test pattern, and a clean `dart pub publish --dry-run`.

### Phase 9.5 — Migration guide and SemVer ADRs (target: 1.0.0 prep) — [#14](https://github.com/willink-oss/willink-design-system/issues/14)

- `docs/adr/0009-semver-policy.md` — define what constitutes MAJOR / MINOR / PATCH after 1.0.0.
- `docs/adr/0010-lockstep-policy.md` — formalize the Flutter-only release exception.
- `docs/MIGRATION-0.7-to-1.0.md` — adopter guide, cross-checked against `git diff v0.7.1 HEAD -- packages/react/src/index.ts`.
- Update `CONTRIBUTING.md` and the `README.md` roadmap section.

### Phase 9.6 — v1.0.0 release cut — [#15](https://github.com/willink-oss/willink-design-system/issues/15)

- Bump the four packages to 1.0.0 in a single release PR with CHANGELOG entries that link to the migration guide.
- Tag `v1.0.0` (npm packages) and `flutter-v1.0.0` (pub.dev) so each workflow publishes from its own source of truth.
- Re-enable `--provenance` in `.github/workflows/publish.yml` now that the repository is public (resolves the note in ADR-0007).
- Draft GitHub Release notes; open a GitHub Discussions post for the announcement.
- File follow-up PRs in consumer repos to bump their dependency range to `^1.0.0`.

## Parallelism

```
                  ┌─ 9.2 (a11y / motion)  ─┐
9.1 (API freeze) ─┤                        ├─→ 0.8.0
                  └─ 9.4 PR1 (TabBar)     ─┘     │
                                                 ├─ 9.3 (Storybook)        ─┐
                                                 └─ 9.4 PR2 (BottomSheet)  ─┤
                                                                            ├─→ 0.9.0
                                                  9.4 PR3 (SnackBar)       ─┤
                                                  9.4 PR4 (Progress)       ─┤
                                                  9.5 (migration / ADRs)   ─┤
                                                                            └─→ 9.6 → v1.0.0
```

- **Strictly sequential:** 9.1 → 9.2 (motion token promotion is a prerequisite); 9.5 → 9.6 (migration doc must land before the tag).
- **Parallel-safe:** 9.2 ‖ 9.4-PR1; 9.3 ‖ 9.4-PR2.

## How to contribute

- Each phase is a GitHub Issue with a checklist. Pick one up, open a PR that references the issue, and tick boxes as you go.
- Keep `pnpm guardrails` green on every PR.
- Storybook PRs must include stories in the same PR — no empty tooling commits.
- Flutter-only PRs bump `willink_theme` alone; do not touch the npm packages unless tokens or preset CSS change.
- Open a GitHub Discussion if a deferred item (dark mode, WordPress, FormField, etc.) needs to be reconsidered for v1.0 scope — proposals are welcome.
