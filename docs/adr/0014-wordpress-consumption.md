# ADR-0014: WordPress / legacy-toolchain consumption — `css-tokens` is the vehicle, no dedicated WP package

- **Status**: Accepted
- **Date**: 2026-06-11
- **Phase**: v1.3 cycle (WordPress consumption)

## Context

"WordPress package" has sat on the roadmap as a deferred consumer-group item since v1.1 ("needs a real consuming theme to pilot against" — v1.2 roadmap). The v1.3 cycle ran that pilot, CEO-approved scope, against two real WordPress consumers of `@willink-labs/css-tokens`:

- **`wp-modern-starter-kit`** — the canonical integration pattern: a modern WP block-theme build (PostCSS toolchain) importing the DS variables wholesale. Merged.
- **`esperanza-wp-theme`** — the production theme that motivated the package in the first place (ADR-0008 stage 2, ADR-0009). Merged.

The pilots proved the package works in WordPress — and surfaced exactly two defects in its distribution shape:

1. **BLOCKER — `exports`-map-only subpaths are unresolvable by plain-path consumers.** The package exposed its files *only* through the package.json `exports` map (`"./tokens.css": "./src/tokens.css"`). postcss-import — and therefore the Tailwind v3 CLI and most classic WP PostCSS toolchains — resolves `@import "@willink-labs/css-tokens/tokens.css"` as a **literal file path** inside `node_modules` and never consults the `exports` map. The documented specifier did not exist as a physical file, so both pilots had to import the physical `src/tokens.scale.css` path as a workaround — an undocumented internal path that the `exports` map, ironically, would have *blocked* under an `exports`-aware resolver.
2. **MISSING — no color-free primitives file.** Esperanza's DS contract is **radius/motion only**; the theme keeps its own palette. The smallest importable unit carrying the radius/duration/easing vocabulary was `tokens.scale.css`, which also ships the full color scales and shadows — the pilot was ingesting **50 inert color/shadow variables** to use 10.

## Decision

### 1. `css-tokens` is the WordPress consumption path — the "WordPress package" roadmap item closes

No dedicated `@willink-labs/wordpress` package will be built. The pilot evidence says the existing framework-agnostic CSS-variables package, with the two distribution fixes below, **is** the WordPress integration:

- **esperanza-wp-theme (production)**: 10/10 variable parity against its hand-maintained SCSS copy; dist output **byte-identical** before/after the migration except for the inert variable additions; 4/4 visual-regression checks pass. The ADR-0008 stage-1 "manual SCSS copy" era is over.
- **wp-modern-starter-kit (canonical)**: documents the reference wiring (npm dev-dependency → PostCSS import → enqueued stylesheet) that any future WP consumer copies.

A dedicated WP package would have nothing left to contain: no PHP runtime is needed (the CSS is enqueued like any stylesheet), and the token vocabulary is already framework-agnostic by construction.

### 2. Root-level proxy files — the postcss-import / `exports`-map lesson

Every documented subpath now also exists as a **physical file at the package root** (`tokens.css`, `tokens.scale.css`, `tokens.semantic.css`, `tokens.dark.css`, `tokens.primitives.css`), each containing a single `@import "./src/<same>.css";`. Generated by `scripts/generate.mjs` alongside the `src/` outputs, committed, whitelisted in `files`, and covered by tests (fs-existence is itself the contract — no resolver involved).

The `exports` map is **kept**, with its entries now pointing at the root proxies, plus a `./src/*.css` passthrough so the pilots' physical-path workaround imports stay valid under `exports`-aware resolvers too. Lesson recorded for any future CSS-shipping package: **a CSS subpath that only exists in the `exports` map does not exist for the PostCSS half of the ecosystem.** Ship the physical file; treat the map as an addition, not the mechanism.

### 3. The primitives-only contract — `tokens.primitives.css`

New granular export carrying **radius + duration + easing only** (10 variables) — zero `--color-*`, zero `--shadow-*` (shadows embed baked-in rgba color and are excluded deliberately). This is the formalization of esperanza's pilot contract — *pilot plan Phase 4.2.1, option (A): primitives-only file in the DS package* — chosen over (B) consumer-side filtering, which every WP consumer would have had to reinvent. Generated, committed, root-proxied, and guarded by a test asserting zero `--color-`/`--shadow-` occurrences with the variable count derived from `primitive.json`.

### 4. Verification responsibility — split by repo

This repo verifies the **package surface**: generator output, proxy resolution, zero-color guard, `files` whitelist (the tests in `packages/css-tokens/__tests__/generated.test.ts`, per [ADR-0012](./0012-release-verification-policy.md) Layer 0/1). The **consumer-side** checks — esperanza's variable parity, dist byte-comparison, and visual-regression suite — live and run in the `esperanza-wp-theme` repo under the CEO's flow, not here. A DS release PR does not re-run or restate them; it links them.

## Consequences

- WordPress is now a first-class consumer group of the existing lockstep npm set — zero new packages, zero new publish workflows, zero new version lines.
- The roadmap sheds its longest-deferred item by *closing* it rather than building it; future WP consumers onboard by copying wp-modern-starter-kit's wiring.
- Plain-path resolvability is now part of the published contract: removing a root proxy, or renaming a file under `src/`, is a breaking change per [ADR-0010](./0010-semver-policy.md) — the test suite enforces presence.
- Consumers with their own palette have a zero-leak import (`tokens.primitives.css`); the 50-inert-vars pattern from the pilot does not repeat.
- The generator now writes outside `src/` (package root). Acceptable: the proxies are one-line, content-stable, and diff-visible in every PR like the rest of the generated output.
